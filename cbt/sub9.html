9. UI 테스트 (상세)

사용성(Usability), 테스트 기법, 결함 관리에 대한 상세 내용입니다.

1. UI 테스트 개요 및 사용성 (Usability)

A. UI 테스트의 목적

  • 사용자에게 결함 없는 기능을 제공하고, 사용의 편의성(Usability)을 높여 만족도를 개선하는 것.
  • ISO/IEC 9126 품질 특성: 기능성, 신뢰성, 사용성, 효율성, 유지보수성, 이식성.

B. 제이콥 닐슨(Jakob Nielsen)의 10가지 사용성 원칙 ★

사용자가 시스템을 쉽게 사용할 수 있는지 평가하는 10가지 기준입니다.

  • 시스템 상태 가시성 (Visibility of system status): 현재 무엇이 진행 중인지 사용자에게 알려야 함 (예: 로딩 바, 배터리 잔량).
  • 시스템과 실세계의 일치 (Match between system and the real world): 사용자가 이해하는 언어와 개념 사용 (예: 휴지통 아이콘, 폴더).
  • 사용자 제어와 자유 (User control and freedom): 실수로 수행한 작업을 취소(Undo)하거나 다시 실행(Redo)할 수 있는 비상구 제공.
  • 일관성과 표준 (Consistency and standards): 같은 기능은 같은 버튼/위치에 있어야 함 (예: '확인' 버튼은 항상 오른쪽).
  • 오류 방지 (Error prevention): 오류가 발생하기 전에 미리 방지 (예: 검색어 자동완성, 삭제 시 확인 창).
  • 기억보다는 인식 (Recognition rather than recall): 사용자가 기억하게 하지 말고 눈에 보이게 함 (예: 메뉴 아이콘, 최근 검색어).
  • 유연성과 효율성 (Flexibility and efficiency of use): 초보자와 전문가 모두를 위한 단축키나 옵션 제공.
  • 심미적이고 최소화된 디자인 (Aesthetic and minimalist design): 불필요한 정보는 줄이고 핵심에 집중 (심플한 디자인).
  • 오류 복구 지원: 오류 메시지는 이해하기 쉽게 쓰고 해결책 제시 (예: "404 Error" 대신 "주소를 확인해주세요").
  • 도움말과 문서화: 사용법을 쉽게 찾을 수 있는 도움말 제공.

2. UI 테스트 기법

A. 전문가 기반 테스트 (Inspection)

사용자가 아닌 UI/UX 전문가가 평가하는 방법입니다.

  • 휴리스틱 평가 (Heuristic Evaluation): 전문가(3~5명)가 '10가지 사용성 원칙'을 기준으로 체크리스트를 보며 문제점을 찾음.
  • 인지적 워크스루 (Cognitive Walkthrough): 학습 용이성을 평가. "사용자가 이 버튼을 보면 무슨 기능을 하는지 알까?"를 시뮬레이션.

B. 사용자 기반 테스트 (Usability Testing)

실제 사용자를 대상으로 수행합니다.

  • 랩(Lab) 테스트: 통제된 환경(실험실)에서 카메라 등으로 관찰하며 테스트.
  • 현장 테스트: 사용자가 실제 사용하는 장소(집, 회사)에서 테스트.
  • A/B 테스트: 두 가지 시안(A안, B안)을 비교하여 더 나은 것을 선택.

3. 테스트 케이스와 시나리오

A. 테스트 케이스 (Test Case)

  • 특정 요구사항을 준수하는지 확인하기 위해 입력값, 실행 조건, 기대 결과를 명시한 문서.
  • 구성요소: 식별자(ID), 테스트 항목, 입력값(Input), 기대 결과(Output), 선수 조건(Prerequisite).

B. 테스트 시나리오 (Test Scenario)

  • 여러 개의 테스트 케이스를 동작 순서(Flow)에 따라 묶은 절차서.
  • 예: [로그인] ➜ [상품 검색] ➜ [장바구니 담기] ➜ [결제]

C. 테스트 오라클 (Test Oracle)

  • 참(True): 모든 입력값 검사.
  • 샘플링(Sampling): 일부만 검사.
  • 휴리스틱(Heuristic): 일부 + 나머지는 직관(추정).
  • 일관성(Consistent): 변경 전후 결과 비교.

4. 결함(Defect) 관리

A. 결함 심각도 (Severity)

  • 치명적(Critical): 시스템 다운, 데이터 손실 (테스트 진행 불가).
  • 주요(Major): 주요 기능 장애.
  • 보통(Normal): 일반적인 기능 장애.
  • 경미(Minor): 화면 글자 깨짐, 오타 등 사소한 문제.

B. 결함 상태 (Status)

  • Open: 결함 보고됨.
  • Assigned: 개발자에게 배정됨.
  • Fixed: 개발자가 수정함.
  • Closed: 테스터가 재확인(Retest) 결과 해결됨.
  • Deferred: 수정을 나중으로 미룸.
📝 시험 전 체크리스트 (Self-Test)
  • 사용자가 실수로 파일을 삭제했을 때 '되살리기' 기능을 제공하는 것은?
    ➔ 사용자 제어와 자유
  • 오류 메시지를 "Error Code: 0x001" 대신 이해하기 쉽게 표시하는 것은?
    ➔ 오류 복구 지원 / 시스템과 실세계의 일치
  • 전문가가 사용성 원칙 기준을 가지고 UI를 평가하는 기법은?
    ➔ 휴리스틱 평가
  • 테스트 케이스들을 실행 순서에 맞춰 묶어 놓은 문서는?
    ➔ 테스트 시나리오
  • 개발자가 수정을 완료했으나 아직 테스터가 확인하지 않은 상태는?
    ➔ Fixed
